## 1. 60 分タイムテーブル

### 0-5 分 関係性温め（孫正義 山の宣言）

開口一番、雑談を挟まずに 1 文で山を開示する。「AI がコンサルタントの思考プロセスを構造化し、中堅中小企業でも大企業品質の戦略立案を実費ゼロで得られる世界を作っています」と宣言し、以後の会話の評価軸を水野氏に先制して渡す。これは孫正義の「交渉の先手は山の定義」の応用。

トーン設定: 主語を「私たちは」ではなく「これが起きている世界は」に統一し、個人営業でなく社会変化の記述として話す。

### 5-15 分 Huang テスト 3 点（Go/No-Go ゲート）

3 点を水野氏と一緒に検証する形で問う。

- 問 1「5 年後もコンサル業は人手不足か」答えを相手に言わせる
- 問 2「大企業コンサルの型を SME に届ける技術的障壁は今消えつつあるか」Claude Code 実演で即回答
- 問 3「類似プレイヤーが既に規模化しているか」最大の審査ポイント（後述 Q3 と連動）

3 点全て水野氏が「なるほど」と言えない場合は佐藤裕介 PL に進まず、問 2-3 の深掘りに留まる。

### 15-30 分 佐藤裕介 3 変数交点 + PL + アセット帰属

3 変数が今この瞬間に重なっていることを 1 枚マトリクスで示す。1 つでも欠けると「良いプロダクトだが負ける」になる理由を対比表で。

- 技術変化: Claude Code MCP / Opus 4.7 / SDK 化（FACT）
- ユーザー変化: 中堅中小事業者の AI リテラシー急上昇（INFERENCE）
- 市場構造変化: コンサル人月単価の高止まり（FACT・公開料金体系より）

PL 骨格の開示シーケンス:
- 楽観 / 基本 / 悲観の 3 シナリオを見せる
- 「銀行審査部出身の水野氏が最初に疑う根拠のない楽観数値を自分たちは見せない」と宣言（consulting-playbook 銀行審査部 3 共通点適用）
- アセット帰属診断: 顧客がプラットフォームを使い込むほど ConsultingOS の知識資産（evolution-log・スキル群）が蓄積される構造を説明

### 30-40 分 Dorsey 削ぎ落とし + Ask

1000 万円の用途を 1-2 点の必達目標に絞って提示。曖昧な「マーケ費」「人件費」を排除し、「軸 A 投資助言業登録 + クライアント環境向け SaaS β 実装、これ以外は使わない」と宣言する。

Ask: 金額と条件を明示。「これを今日合意するには何が必要か」を水野氏に問い返して MTG を一方通行にしない。

### 40-55 分 Q&A 想定問答

後述「2. 想定 Q&A 10 問」参照。

### 55-60 分 次回アクションの合意

3 パターンを想定し、いずれの場合も Next Action を相手から言わせる（「○月○日までに条件書を送る」等）。沈黙で終わらせない。出口は「合意 / 持ち帰り / 再協議条件の明示」の 3 択で水野氏に選ばせる。

## 2. 想定 Q&A 10 問（水野氏 / 銀行審査部出身）

各回答は能動形 + FACT / INFERENCE / SPECULATION ラベル付き。

### Q1 OSS で配っているなら誰でも真似できるのでは

ConsultingOS のコードは公開可能な構造を持つが（FACT）、5 年間の一次情報蓄積（evolution-log・スキル群・顧客案件ログ）が差別化の本体であり、コードを取得した翌日に同等の判断品質を出すことはできない（INFERENCE）。佐藤裕介「5 年蓄積前提の業界精通」原則を適用。コードは手段、蓄積資産が本体。Activity Map 観点では CLAUDE.md / ICP.md / DESIGN.md / 反証ゲートが相互依存しており、1 ピース単独の模倣では機能しない構造（INFERENCE）。

### Q2 Anthropic 自身が同じものを作るのでは

Anthropic は LLM の研究・開発・API 提供が本業であり（FACT）、特定業種・地域・SME 向けのドメイン特化 ConsultingOS を内製する動機は現時点で観測されていない（INFERENCE）。NVIDIA CUDA 原則に照らすと ConsultingOS は Claude Code という基盤の上にしか成立しない設計であり、Anthropic との利益相反は構造上発生しにくい。むしろエコシステム拡張プレイヤーとして機能する（INFERENCE）。

### Q3 agency-swarm（4.3k Stars）等の競合があるのでは

GitHub 上の高 Stars リポジトリの存在は認識している（FACT・agency-swarm 4.3k Stars 2026-05-04 検証済）。汎用 agent フレームワークとドメイン特化 ConsultingOS は用途が異なる（INFERENCE）。Stars 数は開発者コミュニティの関心を示すが SME 向け有償サービスとしての収益実証ではない（FACT）。差異 3 点: コンサル・事業会社向けの業務規律（反証ゲート / ICP 統合 / セキュリティ多層防御）/ 6 部門 30 エージェントの垂直統合設計 / 即日導入（install.sh）vs エンジニアリング工数。直接競合の有無は調査継続中であることを正直に開示（Fess Up Early 適用）。

### Q4 コンサルは人がやる仕事

高度判断・最終責任・クライアント関係構築は今後も人が担う（INFERENCE）。ConsultingOS が置き換えるのは情報収集・型化・ドキュメント生成・KPI 追跡であり、コンサルタントの稼働時間の大部分を占める低付加価値作業の代替（INFERENCE）。「人ならではの価値が強まる構造」と「AI 代替が進む構造」は並立する（FACT・既に複数調査機関が同様の構造を指摘）。

### Q5 投資助言業登録の不認可リスクは

投資助言業の対象となる「有価証券に関する助言」は提供サービス範囲に含まれていない（FACT・現在の事業スコープ）。経営戦略・マーケティング・事業計画支援は金商法の投資助言業には該当しない（FACT）。軸 A の金融 AI 環境構築支援も「環境構築・教育・ダッシュボード提供」の登録不要範囲で先行受注可能（INFERENCE / 弁護士確認待ち）。登録は Phase 2 並行で 6-13 ヶ月想定（INFERENCE / 金融庁公式資料ベース）。将来的な機能拡張時は legal-compliance-checker による事前審査を義務付け（FACT）。

### Q6 1 名兼務体制は属人化では

現状 1 名兼務であることは事実（FACT）。ConsultingOS の設計思想は「人ではなく構造が動く」であり（FACT）、CLAUDE.md・スキル群・evolution-log を参照すれば別の担当者が同水準のアウトプットを出せる状態を目指している（INFERENCE）。1000 万円の用途には「2 名目の担当者オンボーディング」を含めない（Phase 3 以降）。代替案: 顧問契約 or 業務委託で 4 思想分散（水野氏の要求次第）。

### Q7 1000 万円使い切ったらどうする

1000 万円の設計は「次の資金調達 or 黒字化のどちらかを達成するためのランウェイ確保」が目的（INFERENCE）。標準シナリオで月次固定費 30 万 / ブレイクイーブン 7-10 ヶ月目（INFERENCE）。最小構成（フルリモート + 役員報酬ゼロ + ジュニア不採用）なら 67-83 ヶ月のランウェイ確保可能（INFERENCE）。撤退判断の数値基準は事前定義済（KSF-3「比率 50% 未満 2 ヶ月連続 = 固定費削減」）。

### Q8 3 年後の出口戦略は

3 パターンを提示し水野氏に選ばせる。
(a) 有償顧問 ARR の自己増殖（ブートストラップ・推奨）
(b) 戦略系コンサル / SaaS 企業への事業売却
(c) 独立上場（SPECULATION・現時点では蓋然性評価困難）
水野氏に「どのパターンが最もリターン設計と合致するか」を問い返し、一方的な宣言にしない。

### Q9 私が出資するメリットは何か

3 点を能動的に提示。
(1) SME 向け AI コンサル市場の初期参入ポジション（INFERENCE）
(2) エンジェルとしての情報優位 — コンサル業 × AI ツール化の一次情報が継続的に水野氏に流れる構造（FACT・ConsultingOS の顧客事例は出資者と共有する設計）
(3) 水野氏の銀行審査部知見を ConsultingOS の SME 融資支援機能として取り込む相互価値（INFERENCE）

### Q10 他にも出資検討者はいるのか

現在の検討状況を正直に開示する（Fess Up Early 規律）。「他に X 名と話している」「このラウンドは水野氏が最初の打診」等、実情を隠蔽しない。過剰な希少性演出は信頼毀損リスクが高い（pr-communications.md 隠蔽提案を反証モードで自動却下適用）。条件変更の可能性がある場合はその旨も明示する（FACT として開示できる範囲のみ）。

## 3. MTG 後の対外コミュニケーション設計（Dorsey Fess Up Early 規律）

### 合意成立時

当日または翌日に feature ブランチ（feature/mizuno-angel-round-2026-05）を切り、合意内容のサマリーを commit する。main への Squash and merge 後に短文プレスリリース草稿を準備（配信は法務確認後）。

プレスリリース構造:
- 見出し「AI コンサル OS ConsultingOS、エンジェルラウンド合意」
- リード「2026-05-XX、ConsultingOS は水野氏との資本業務に合意した」
- 詳細は合意範囲内のみ記述

合意後 48 時間以内を目安に note または X で同内容のソートリーダーシップ記事を配信し、信頼の蓄積に変換する（pr-communications.md アーンドメディア優先原則）。

### 部分成立時（条件付き合意）

条件クリアまでの間、対外発信は封印する。ただし「封印」は沈黙ではなく、note / X で業界論考・事例分析の寄稿を月 1 本ペースで継続する（pr-communications.md ソートリーダーシップ構築シナリオ適用）。信頼残高を積み上げながら条件充足を進めることで、再提案時の信頼基盤が高まる（INFERENCE）。

### 不成立時

短文で事実だけを開示する。「当ラウンドは成立しなかった」1 行のみ。過度な弁明・理由の詳細説明・攻撃的発言は禁止（pr-communications.md 謝罪 → 事実開示 → 解決策の 3 セット適用、ただし謝罪フェーズは不要）。evolution-log に「5/7 不成立 / 構造的失敗の理由」等を記録（evolution-log 規律適用）。audit 100 達成の数値証拠が揃い次第、再協議を申し入れる布石として「audit 100 達成時に改めてご報告します」と 1 文添える。

## 【反証チェック結果】

Step 1（自己反証）: 孫 → Huang → 佐藤 → Dorsey の順序に必然性があるか。「山」を先に出せば水野氏が Huang テストを拒否するリスクも存在（反証成立）。対策: Huang テスト 3 問は「一緒に検証する」対話形式にし、審査ではなく共同探索として設計。

Step 2（構造反証）: Q&A の Fess Up Early 適用が「過剰開示」になるリスク。水野氏に不利な情報を自発的に提示しすぎると投資意欲を損なう可能性（反証成立）。対策: 開示対象は「隠蔽すると後で発覚した際の信頼損失が大きいもの」に限定、戦略的曖昧性（競合調査中の旨）は維持。

Step 3（実用反証）: 60 分タイムテーブルは相手の反応速度に依存。Huang テスト 3 点で議論が白熱した場合、佐藤裕介パートに時間が到達しない可能性。対策: タイムキープ役を assistant 側で保持、15 分経過時に強制移行。

残存リスク:
- 「3 年後の出口戦略」Q8 で (c) 独立上場 SPECULATION の判断が水野氏の期待値設定を誤らせる可能性。strategy-lead 出力との齟齬を MTG 前再確認。
- 4 思想展開順序が水野氏の質問パターンに合わない場合の即時切替（Dorsey 前倒し or 佐藤裕介省略）の判断軸が assistant 側で未明文化。
