# 設計起源 — なぜこの構造なのか

## 出発点

Sarvesh Shrivastava 氏（Favikon SEO世界ランキング1位、TEDxスピーカー、Forbes 掲載）が
14年間のローカルSEO実務から構築し、594万ビュー・16,000ブックマークを獲得した
「20 SEO Design Patterns」の記事から、SEO特化を取り除いて**メタ構造**だけを抽出した。

## 抽出したメタ構造（3層）

### 1. Step 0 = 文脈ロード（CLAUDE.md ＋ intake）

> 初対面のコンサルタントに何も説明せず「SEO改善して」とだけ言えば、的外れなアドバイスが返ってくる。AIも全く同じ。

事業情報を6カテゴリ（事業基本／市場／目標／現状数値／競合／作業方針）で一度だけ渡す。
これだけで以降の全出力が「あなた専属のアドバイザー」モードに切り替わる。

実装: `CLAUDE.md` ＋ `templates/intake/client-context.md`

### 2. 20パターン = Skills（再利用可能な業務手順）

属人化を排し、誰がいつ実行しても同じ品質が出るよう「業務パターン」を SKILL.md として定義。
本リポジトリではSEO特化のパターンは取り込まず、業界横断で使える5スキルに絞った。

| 元パターン群 | 抽象化後 |
|---|---|
| GBPカテゴリ監査・属性監査・競合レビュー分析 | `competitor-scan` |
| キーワードギャップ・コンテンツギャップ | `gap-analysis` |
| 検索意図マッピング（4段階） | `intent-mapping` |
| 月次パフォーマンスレポート | `monthly-impact-report` |
| （全パターンの前提） | `client-context-load` |

### 3. 「Claude が作業する／人間が判断する」

各スキルは**結論を出さない**。優先度のシグナル・象限分類・選択肢の提示までに留め、
「どこに投資するか」「どのリスクを許容するか」は必ずクライアント／コンサルタント側に残す。

## なぜSEO特化を取り込まなかったか

- 本リポジトリは SEO ツールではなくコンサルティング基盤である
- SEO 特化の語彙（GBP、被リンク、エンティティ）は intake 側で渡せばよい
- メタ構造だけ採用すれば、営業／採用／バックオフィス／プロダクト戦略にも転用可能
