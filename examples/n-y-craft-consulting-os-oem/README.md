# 実例：N.Y.CRAFT × ConsultingOS OEM 提案 — 第 1 リファレンス案件

ConsultingOS（30 エージェント × 19 スキル × 6 部門 + 反証モード + 業界プレイブック）を、
他社にカスタマイズ提供する **OEM 第 1 号案件** の提案フェーズ記録。

属人化された経営知見を「規律ある専門家チーム」として再現するパッケージとして、
N.Y.CRAFT 株式会社（代表：関根優氏）に提案。提案 PDF を 2026-05 にクライアント送付済。

## 使用したエージェント・スキル・ツール構成

### コア構成

```
[ メインエージェント（オーケストレーター） ]
            │
            ├── 戦略フレーム適用（佐藤裕介・小野寺信行思想）
            │
            ├── スキル
            │     consulting-playbook（提案・戦略の標準手法）
            │     brand-guidelines（日本語字形禁則 / 出力規律）
            │     falsification-check（反証モード Step 1-3）
            │
            ├── ハードルール
            │     #10 日本語字形検証必須（pdffonts機械検証）
            │     #16 出力フォーマット規律（** 禁止 / 表中央 / ページ収まり）
            │
            └── ツール
                  Marp CLI → HTML / PPTX / PDF 生成
                  fontconfig + Noto Sans CJK JP（日本語フォント）
                  pdffonts（埋込フォント機械検証）
                  generate.sh（生成 + 検証 ワンショット）
```

### 思想的基盤の活用

- **佐藤裕介フレーム**：価格決定（4 軸：支払い能力・価値・コスト・代替）／撤退基準先決め／PL 試算ハードルール／敵地で戦わない
- **小野寺信行式**：指標を疑う／フロー × ストック統合（業務委託 = フロー、OS = ストック資産）

## ワークフロー記録（v1 → v7 イテレーション）

提案 1 通の作成に **7 回のイテレーション** が発生。各回でユーザーフィードバックに即応してアウトプットを進化させた記録。

### v1 — 副業 OEM 戦略の汎用枠組み

ユーザー指示：「コンサル起動、佐藤裕介判断に任せる」「副業路線で」
出力：`strategy/oem-side-business/` に汎用 OEM 副業戦略を策定。
学び：先に汎用フレームを作ると、後の個別案件への適用がスムーズ。

### v2 — N.Y.CRAFT 案件用パッケージ初版

ユーザー指示：「html→ppt 出力もするべきでは？」「Marp で行きます？」「うん」
出力：5 ファイル（README / 戦略 / 提案 / 価格根拠 / 打合せプレイブック）＋ Marp スライド 23 枚。
学び：Marp は markdown → PPTX/PDF/HTML の決定論レンダリングで CI 親和性高い。

### v3 — 副業時給フロアからの値段見直し

ユーザー指示：「レンジ高い気がするな」「考え方と根拠が分からないと」
出力：価格 100 万 → 70 万に引下げ、4 軸（支払い能力・価値・コスト・代替）で再構成。
学び：価格は感覚で出すと外す。4 軸の交差点で論理化するのが佐藤裕介流。

### v4 — OS 進化を踏まえた全面練り直し

ユーザー指示：「コンサル OS を進化させたから踏まえて練り直してみて」
出力：30 エージェント × 19 スキル × 6 部門 + 反証モード + 業界プレイブック を反映、価格 70 万 → 100 万に引上げ。
学び：OS が進化したら提案も同期して再構築する。スキル定義書を必ず提案に反映する。

### v5 — 日本語字形違反の修復

ユーザー指示：「日本語の漢字が中国語みたいになってる、スキルを無視してない？」
出力：レンダリング環境の WenQuanYi 削除、`fonts-noto-cjk` 導入、fontconfig で JP 変種を強制プリファ、`pdffonts` で機械検証。
学び：CSS の font-family 指定だけで満足するな。レンダリング環境のフォント有無まで確認し、`pdffonts` で埋込実体を機械検証する（OS 規律 #10 の起点）。

### v6 — リテラシー対応＋ Claude Code との差別化

ユーザー指示：「関根さんのリテラシーに合わせて、RAG や VOC などは注釈つけるか分かりやすく」「コンサル OS のイメージがわかない」「これがなぜ普通のクロードコードと違うのか何がすごいかが分からない」
出力：専門用語を初出のみカッコ書きで日本語化、「普通の Claude Code との違い」スライド追加、「思想的基盤」スライド追加（30 名のエージェントが拠って立つ知見）、ストーリー 2 を「新人立ち上げ」→「関根さん自身の中小企業 A 社案件」に書き換え。
学び：クライアントの業界・リテラシー文脈を読み、エンジニア向け用語を業務文脈に翻訳する。提案は「価値訴求 → 構造 → 価格」の順。

### v7 — リファレンス位置付けの裏方化

ユーザー指示：「リファレンスに関しては小野寺側は裏方だからこちら側は何もしない、あくまで関根さんの会社としてリリースできる」
出力：「お願いする協力」列を削除し、N.Y.CRAFT 単独で発信できる構造に再構成。「当社（提供者）は裏方に徹し対外的には一切表に出ません」を明示。
学び：リファレンス案件＝双方の協力ではなく、クライアント側の発信自由度として位置付ける選択肢もある。等価交換の構図にこだわらない。

## 規律違反と是正の記録（OS にフィードバック）

提案作成中に発生した規律違反 → 検出 → 修正のループ。各ケース main 反映済み（OS 規律 #10/#16 の起点）。

| 違反内容 | 検出方法 | 是正 | OS 学習 |
|---|---|---|---|
| 中国字形フォールバック | `pdffonts` で WenQuanYiZenHei 検出 | fontconfig で SC/TC/HK/KR をリジェクト | 規律 #10 「pdffonts 機械検証必須化」追加 |
| Markdown ** 太字濫用 | `grep '\*\*'` で多数検出 | HTML `<strong>` または「」に置換 | 規律 #16 「** 禁止」追加 |
| 表が中央揃え未指定 | 視覚確認 | CSS で全表に `margin: 0 auto + text-align: center` | 規律 #16「表中央揃え必須」 |
| ページ収まり未検証 | スライド画像化で検出 | 密集スライドのフォント縮小・密度調整 | 規律 #16「PPT/PDF ページ収まり必須」 |
| 一文中改行（句読点直前/直後 br） | `grep '、<br>'`等 | `<br>` を意図的レイアウト位置に限定 | 規律 #16「一文中改行禁止」 |

## 学び

### Sato 判断フレームの提案実装パターン

1. **価格設計は 4 軸の交差点**：支払い能力・価値・自分のコスト・代替手段の重なる帯で決める。単軸で出すと外す。
2. **撤退基準を先に決める**：「いくらまでなら受けない」を契約交渉前に明文化。値切られても折れない。
3. **副業の年間トータル時給で見る**：プロジェクト単体時給がフロア割りでも、業務委託＋ランニング＋成果報酬の年間合算でフロア超えるなら成立。
4. **既存関係はキャッシュフロー設計の一部に組み込まない**：本提案外として明示。混ぜると後で揉める。
5. **リファレンス案件は等価交換にこだわらない**：クライアント単独発信＝こちら裏方に徹する選択もある。

### スライド設計の Sato ルール

1. **価値訴求 → 構造 → 価格の順**：金額を最後に出す。順序を間違えると価格でドン引きされる。
2. **Before / After ストーリーで「動き」を見せる**：機能列挙ではなく具体的な業務シーンの変化を描く。
3. **クライアントのリテラシーに合わせて専門用語を初出のみカッコ書き**：RAG、VOC、ICP.md、evolution-log 等は1 度説明、以降クリーン使用。
4. **比較スライドで「敵地で戦わない」**：強い敵（ChatGPT 全方位比較）は避け、弱い敵（NotebookLM の RAG 限界）に絞る。
5. **ベンダー名・OS 名の開示義務なしを明示**：クライアント側の広報自由度を担保。

### 日本語字形・出力品質ガード

1. **`pdffonts` でフォント機械検証必須**：CSS 指定だけでは不十分。レンダリング環境のフォント有無まで確認。
2. **fontconfig で JP 変種を強制プリファ**：SC/TC/HK/KR をリジェクトする conf を `/etc/fonts/conf.d/` に置く。
3. **WenQuanYi 等の中国系 CJK フォントは削除**：レンダリング環境から物理的に排除。
4. **`generate.sh` のような検証つきヘルパースクリプト化**：手動検証の漏れを構造的に防止。
5. **OS 規律 #16 の 5 項目セルフチェック**：** 不在 / 一文中改行なし / 表中央揃え / ページ収まり / W チェック実施。

## 再現手順（同種案件のテンプレート）

1. **クライアント関係性の評価**：既存業務委託 / 知人紹介 / コールド の 3 段階で初期インタビュー深度を判定
2. **副業フレーム判定**：`strategy/oem-side-business/03-ideal-partner-profile.md` の Green/Red ゾーンで受諾可否
3. **戦略フレーム策定**：`01-strategy.md` テンプレートで案件位置付け・撤退基準を先決め
4. **価格根拠の 4 軸算出**：`03-pricing-rationale.md` テンプレートで支払能力・価値・コスト・代替の交差点
5. **打合せ運営台本**：`04-meeting-playbook.md` テンプレートで提示順序・想定問答・NG 対応
6. **提案 PDF 制作**：Marp slides.md ベース、`generate.sh` で生成 + 機械検証
7. **クライアントフィードバックループ**：v1 → v7 のように小刻みに修正、各回で機械検証通過を必須
