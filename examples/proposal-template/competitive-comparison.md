# 競合比較表 設計ルール（P8）

> hotice deck 実装で確立した「敵地で戦わない」競合比較の設計指針。ポーター式 5フォース + 業界構造の収益性上限 + ティール式 10x 優位 をベースに作る。

---

## 設計原則（最優先）

### 1. 載せる軸の選定
✅ 載せる:
- 自社が勝つ軸
- 全社で差分が出る軸（自社のポジションが読み取れる）
- ポジションマップが自然に成立する軸

❌ 載せない:
- 「非公表 / 要相談」が半数以上になる行
- 自社が負ける軸
- 自社が不利な領域での横並び比較

### 2. 商品名 < 解釈した強み
❌ NG: 「AnyTag / GOAT / MTA を保有」
✅ OK: 「多国 × AI SaaS による効率化」「インフルエンサー領域での独自データ蓄積」

→ ツール名・商品名ではなく、解釈された強みを主文に書く。

### 3. 領域特化（ポーター式 Focus 戦略）
競合カテゴリによって比較領域を限定する:

| 競合タイプ | 比較領域 | 例 |
|---|---|---|
| 総合広告グループ（電通・博報堂） | 「インフルエンサー領域での」強み | × 総合広告全般での横並び |
| インフルエンサー特化（UUUM・BitStar） | 「クライアント側ニーズ理解」 | × クリエイター数の横並び |
| グローバル SaaS（AnyMind） | 「日本市場での運用力」 | × プラットフォーム数の横並び |

→ ポーター式「Focus 戦略」: 全方位で勝とうとせず、勝てる領域に絞り込む。

### 4. 自社弱点を自分から surface しない
- 「留意点」「今後の課題」「改善中」のような行を自発的に作らない
- これらが必要な場合は、別ページで提案改善策とセットで触れる

---

## 比較表の標準フォーマット

### 推奨構造（4-6社、5-7軸）

```
| 軸 | 自社 | 競合A | 競合B | 競合C |
|---|---|---|---|---|
| 領域特化度 | ◎ インフルエンサー専業 | △ 広告全般の一部 | ○ MCN中心 | △ 越境主軸 |
| 実績データ蓄積 | ◎ 過去○年分 | △ 非公表 | ○ 一部公開 | △ 海外中心 |
| 国内代理店連携 | ◎ ○○社実績 | ○ 系列代理店 | △ 直販中心 | × 直販のみ |
| ナーチャリング設計 | ◎ オムニチャネル統合 | △ デジタル単体 | × クリエイター起点 | △ プラットフォーム依存 |
| 計測指標の柔軟性 | ◎ 目的別 KPI 分岐 | △ ROAS 一律 | × 再生数中心 | △ 限定指標 |
```

凡例: ◎ 強み / ○ 同等 / △ やや弱い / × 弱い（自社の × 行は載せない）

---

## ポーター式「業界構造ゲート」前置き

競合比較を作る前に、業界の 5フォースを診断する:

```
【業界構造の前提】
1. 既存競合の競争激度: <強/中/弱>
2. 新規参入の脅威: <強/中/弱>
3. 代替品の脅威: <強/中/弱>
4. 買い手の交渉力: <強/中/弱>
5. 売り手の交渉力: <強/中/弱>
→ 業界の構造的収益性上限: <ROIC X% 程度>
```

業界が構造的に儲からないなら、提案そのものを再考する。

---

## ティール式「独占性ゲート」付記（B2B SaaS の場合）

提案する自社サービスが以下を満たすか確認:

- [ ] 10x 優位: 既存代替案より10倍以上優れているか（漸進改善は警戒）
- [ ] 独占の4要素のうち最低2つ:
  - [ ] Proprietary Technology（プロプライエタリ技術）
  - [ ] Network Effects（ネットワーク効果）
  - [ ] Economies of Scale（規模の経済）
  - [ ] Branding（カルト的選好）
- [ ] Last Mover ポジション: 構造的堀があるか
- [ ] Secret thesis: 「重要だが他社が同意しない真実」を1文で書けるか
- [ ] 小さく明確な市場から始められるか

満たさないなら「敵地で戦う提案」になっている可能性大。再設計を推奨。

## 反証チェック

### コピー
- [ ] 全競合の情報が公式 / IR / 主要紙ソース経由
- [ ] 出典 URL + 発行日が全て紐づいている
- [ ] 競合の弱みを事実ベースで書いている（誹謗中傷でない）
- [ ] 自社の強みは実証データで裏付けられている

### 設計
- [ ] 軸が「自社が勝つ or 全社で差分が出る or ポジション読み取り可」のいずれか
- [ ] 「非公表 / 要相談」半数以上の行がない
- [ ] 自社の弱点が surface されていない（留意点行なし）
- [ ] 商品名ではなく解釈した強みが主文
- [ ] 領域が限定されている（ポーター式 Focus）

### 実用
- [ ] クライアントが「自社の他社優位性」をこの表だけで理解できる
- [ ] 表が読み手の意思決定を促す構造になっている

## hotice deck 実装での教訓

- ❌ 失敗例: 「AnyMind 主戦場 Shopee/Lazada」 → クライアント側で「東南アジア限定？」と誤解されかけた
- ✅ 修正後: 「多国プラットフォーム統合中心」 → 領域は限定しつつ、解釈の強みを残す表現に
- ❌ 失敗例: 「業界随一」「最も数をこなしている」 → 出典なし最上級は信頼を毀損
- ✅ 修正後: 「実績と訴求パターンを熟知している」 → 検証可能な表現に
