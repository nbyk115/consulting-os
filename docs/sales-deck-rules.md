# セールス資料デザイン・コピールール v1.0

2026年4月に `hotice` 社の代理店向けセールス資料を再構成した際の学びを一次ルール化。以後、同種の資料（特に **代理店／エンタープライズ向け提案資料**）を作る際は本書を参照する。

---

## 0. オーケストレーター原則（佐藤裕介モード）

1. **市場構造から入る**。「なぜ今」「どこで稼ぐか」「競合構造」を先に宣言。
2. **断言形、二項対立**。動詞で締める。曖昧なのれん分け表現は禁止。
3. **PLに落とせない提案は却下**。精神論・飾り言葉を載せない。
4. **反証モード常時ON**。全数値にソースURL。未確認なら `NOT VERIFIABLE` として軟化 or 削除。
5. **敵地で比較しない**。自社が負ける軸は載せない。自分のフィールドで勝負する。

## 1. コピーのルール

### 1-1. タグライン / ヘッドライン

| ❌ NG | ✅ OK | 理由 |
|---|---|---|
| ブランドの〈伝えたい〉をユーザーの〈聞きたい〉に翻訳する | "買いたい"を、売上に変える | 抽象語 → 具体語、動詞で締める |
| 〈機能〉を〈体験〉に翻訳する | "売りたい"を"買いたい"に翻訳する | 二項対立が明確 |
| "気づき"を売上に変える | "買いたい"を売上に変える | "気づき"は定義されないと不明瞭 |
| 勝ち筋は「人間の選定」に戻る | 勝ち筋は"量"から"質"、そして"実装力"に | 読者が追える進行を入れる |

**原則**：主語 × 動詞 × 目的 が一文で完結する。修飾形容詞で逃げない。

### 1-2. 禁止表現

- **上から目線**："代理店が…で組めます" → "代理店様との組み方は…。選んでいただけます"
- **無根拠の最上級**：「最も蓄積している」「業界最大級」 → "連続運用の実績と訴求パターンを熟知している"
- **自社のネガティブな言及**："非上場・少数精鋭" は留意点行として自分で出さない
- **ダサい動詞締め**："作った。" "広げた。" → "押し上げる。" "呼び込む。" "届ける。"
- **業界略語の露出**：MCN / MKT / MTA / GOAT など → 日本語で。必要なら小さい注釈で英略語併記

### 1-3. 敬語・トーン

- `代理店様` を使う（`代理店` だと距離が出る）
- `御社` / `貴社` を使い分け（資料文中は `御社`）
- "〜ください" より "〜いただけます"（お願いより提供）
- 業界ヒアリングやクライアント固有データは **匿名化**（上場企業はフェアディスクロージャー配慮）

### 1-4. 3-カード / 並列要素

- 全カード同じ行数（1行 / 2行 / 3行で統一）
- 同じ文型で始まる（`○○案件の ○○実装枠` のような並列構造）
- 見出しと大数字で **同じ情報を二重に言わない**（`最短 2 週間` の下に `2週間` は不要）

## 2. ハルシネーション防止ルール

### 2-1. すべての数値にソース紐付け

| 数値カテゴリ | 必須情報 |
|---|---|
| 市場規模 | 調査会社名 + 調査年 + リンク |
| CMO / マーケター サーベイ | レポート名 + 発行元 + 母数 + リンク |
| 競合情報 | 一次ソース（IR / 公式プレス / 主要紙）+ 発行日 + リンク |
| 自社実績 | 期間 + 対象範囲（例：「2021-2024 Amazon アフィリエイト施策の加重平均」） |

### 2-2. 代理店ヒアリング等の非公開情報

- **資料に載せない**（口頭共有のみ）
- 推定値を載せるなら「大手総合代理店ヒアリング・○社調べ」を明示 ← 推奨しない
- 削除して、公表データに置換可能なら置換（推奨）

### 2-3. ケーススタディ

- 実名可 = 一次ソース（PR TIMES / 公式発表）が存在する場合のみ
- それ以外は **匿名化**：「グローバル新業態カフェ」「急成長ウェルネスD2C（上場企業）」
- 脚注「※ クライアント実名はミーティング時に口頭で共有いたします」を追記

## 3. 競合比較表の設計

### 3-1. 行の選び方

**✅ 載せる軸**：
- 自社が勝つ／唯一対応している軸
- 全社で明確な差分が出る軸（稼働マーケット、言語対応、料金モデル）
- ポジショニングが読み取れる軸（事業モデル、強み）

**❌ 載せない軸**：
- 半数以上が "非公表 / 要相談" になる軸
- 自社のネガティブ情報が露出する軸（留意点・弱み行）
- 数字の大小で負ける軸（DBサイズで相手が圧倒的に上の場合）

### 3-2. 「強み」行の書き方

- 2024-2026 の**公式プレスを解釈した派生強み**で書く
- 商品名そのまま（"AnyTag" "UUUM ONE"）は小サブに降格、**解釈した強み**を主文に
- ❌ `AnyTag (MTA 2024 最優秀)` / `UUUM ONE ／ GOAT`
- ✅ `多国 × AI 運用の SaaS` / `トップYouTuber × AI動画`

### 3-3. 数値の文脈化

- 相手の規模が大きい場合は **文脈化して自社の意味を保つ**
- 例：AnyMind DB 100万 / BitStar 累計400万 vs hotice 25,000
  - 軸を `稼働ベース規模` に変更
  - hotice = `稼働 25,000+`（実運用）
  - 相手 = `DB 100万+（ツール上の登録数）` / `累計400万（稼働数は非公表）`
  - → hotice の "稼働" が質的な意味を持つ

## 4. レイアウトのルール

### 4-1. 横方向の揃え（最重要）

- 複数カードを並べる際、**proof boxes / 小さいラベル** が横一線で揃っていること
- グリッドで行を固定：`grid-template-rows: auto auto auto 1fr auto` などで最下段（CTA/Answer）を固定
- 数値ラベル列は **固定幅 + `white-space: nowrap`**（日本語の単位が改行されない）

### 4-2. 縦方向（上下余白）

- 下端に `.srcLine / .footnote / .caseNote` がある場合、`margin-top: auto` で貼り付け
- スライド下部に大きな白帯があるページは、main content を fill させるか下段要素を pin
- `<br>` は **文節境界**で入れる。"マーケティン + グ会社" のような中途半端な改行を避ける

### 4-3. 文字収まり

- `.h1` は `word-break: auto-phrase; line-break: strict; overflow-wrap: break-word`
- 日本語長文が入る要素には 全て 上記を適用
- フォントサイズを下げれば治る場合は下げる（36→34→32）

### 4-4. ページ番号

- **大きい `.pageno` バッジは非表示**。右下の小さい `X / 18` のみ
- 下左は "hotice" のみ（バージョン文字列は非表示）

### 4-5. 禁止の技

- `transform: scale(1.05)` で特定ボックスを拡大 → 行の整列が崩れる
- 数字を `bottom: -12px` で突き抜けさせる → clip されて「キレ」てしまう
- `class="brand"` のようなグローバル衝突する class 名を別用途で使う

## 5. 情報設計（ストーリー）

### 5-1. ページ並び（代理店向け提案の基本形）

```
P1  表紙（ブランド × タグライン）
P2  自社紹介（会社規模・KPI・サービス・主要クライアント）
P3  市場の現在地（数字＋グラフ）
P4  市場の構造変化（時代区分で"なぜ今"）
P5  代理店が直面する論点（現実に沿う／否定しない）
P6  自社ポジショニング（特色・職能）
P7  競合比較表
P8  なぜ自社か（3理由）
P9  差別化1（例：グローバル実装）
P10 差別化2（例：スピード × 品質）
P11 差別化3 / 実績サマリー
P12-15 ケーススタディ（実名可1本 + 匿名3本）
P16 協業パターン（3モード）
P17 料金・体制・納期
P18 CTA（次ステップ）
```

### 5-2. 各ページ 1つの役割

- 1ページで2つ以上のメッセージを混ぜない
- "なぜ" と "何を" と "どう" は別ページ

### 5-3. トーンの軸変化

- P1-2: 静 / 定義
- P3-5: 市場・論点 / 知的
- P6-11: 差別化 / 強気
- P12-15: 実証 / 信頼
- P16-17: 実務 / 透明
- P18: 決断 / 情熱

## 6. 協業モデルの分離（MODE 01 vs 02 vs 03 の教訓）

ホワイトラベル vs 実行支援 vs 共同提案が混ざりやすい。**タイミング × 範囲 × 露出** の3軸で分離する：

| | タイミング | 範囲 | 露出 |
|---|---|---|---|
| 01 共同提案型 | **案件獲得前**（ピッチ段階） | 企画 + 実装 | 連名で開示 |
| 02 ホワイトラベル型（推奨） | 受注後 | 戦略 + 実装 すべて | 不可視・御社名義 |
| 03 実装ベンダー型 | 受注後 | **実装のみ** | 通常のベンダーとして開示可 |

**MOST POPULAR / 推奨ラベル** は必ず意味を説明（なぜ赤強調か）。

## 7. 実装のプロセスルール（Claude Code）

1. **小さく変える → レンダ → 目視 → push**。大量変更を盲目的に commit しない
2. Puppeteer でPNGを生成し、必ず `Read` で目視確認
3. 外部リソース（Unsplash 等）は Puppeteer で読み込まれない可能性が高い → ローカル or SVG / typographic で代替
4. commit メッセージはプッシュごとに具体（何をどう直したか）
5. CDN キャッシュ対策：raw.githack.com の代わりに **rawcdn.githack.com + commit SHA** を貴方に渡す
6. リサーチはサブエージェントに並行委任（本線を止めない）
